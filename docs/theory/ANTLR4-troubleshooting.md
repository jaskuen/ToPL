## 1. Учёт приоритета операторов

Операторы в выражениях имеют разный приоритет (например, `*` выше, чем `+`). Если грамматика не отражает эту иерархию, парсер может строить дерево, не соответствующее математической или языковой семантике.

Выражение `a + b * c` интерпретируется как `(a + b) * c`, что приводит к ошибкам выполнения или компиляции.

Визуализация дерева разбора показывает неправильную вложенность узлов, и тестовые примеры дают неожиданный порядок вычислений.

### Решение

**Использовать многоуровневую иерархию правил**, где каждый уровень соответствует группе операторов одного приоритета.

```antlr
expr     : expr ('+' | '-') term    # AddSub
         | term                     # TermExpr
         ;
term     : term ('*' | '/') factor # MulDiv
         | factor                   # FactorTerm
         ;
factor   : INT                     # IntLiteral
         | ID                      # Variable
         | '(' expr ')'            # ParenExpr
         ;
```

## 2. Левая и правая рекурсия

### Суть проблемы

- **Левая рекурсия**: правило вызывает само себя как первую альтернативу, что приводит к зацикливанию:
  `expr: expr '+' term`.
- **Правая рекурсия**: правило вызывает себя в конце и может привести к неправильной ассоциативности:
  `expr: term '+' expr`.

- Ошибка при генерации:  
  `error(119): The following rule is left recursive [...] and contains actions or predicates...`
- Неожиданная форма дерева (например, правоассоциативное группирование для `-`).

### Решение

- Для **левоассоциативных** операторов используйте **прямую левую рекурсию**:

```antlr
expr: expr '+' term
    | expr '-' term
    | term
    ;
```

## 3. Левоассоциативные операции

### Суть проблемы

Операторы вроде `+`, `-`, `*`, `/` должны группироваться слева направо:  
`a - b - c → (a - b) - c`.

Если группировка идёт справа, результат вычисления неверен:
Например, `10 - 5 - 2 = 3`, а не `10 - (5 - 2) = 7`.

Дерево разбора имеет форму правостороннего дерева


### Решение

Используйте **леворекурсивные правила** — ANTLR4 автоматически строит **левоассоциативное дерево**

```antlr
expr: expr '-' term   // левоассоциативно
    | term
    ;
```

## 4. Правоассоциативные операторы

### Суть проблемы

Некоторые операторы (например, возведение в степень `^`, присваивание `=`) группируются **справа**:  
`a ^ b ^ c → a ^ (b ^ c)`.
Левоассоциативная обработка даёт неверный результат:
`2 ^ 3 ^ 2 = 512`, а не `(2 ^ 3) ^ 2 = 64`.

Дерево разбора группирует слева, хотя ожидалось справа.

### Решение

Явно укажите ассоциативность с помощью **опции `<assoc=right>`**:

```antlr
expr: expr '^' <assoc=right> expr
    | term
    ;
```

ANTLR4 при разборе будет **предпочитать правостороннее дерево** даже при леворекурсивной форме.

> ⚠️ Опция `<assoc=right>` **работает только внутри одного правила** и только при наличии **двух рекурсивных альтернатив**. Не применяйте её к терминальным альтернативам.

## 5. Неоднозначность: переменная vs вызов функции

### Суть проблемы

Выражения вида:

- `x` — переменная
- `x()` — вызов функции без аргументов

имеют общий префикс (`ID`), что создаёт **локальную неоднозначность** на этапе предпросмотра. Парсер может выбрать неправильную альтернативу и возможны предупреждения `non-LL(*) decision` или `ambiguity`.

### Решение

#### Вариант A: Порядок альтернатив + жадность

ANTLR4 использует **жадную стратегию** (longest match). Поэтому **более длинная альтернатива должна идти первой**:

```csharp
primary: ID '(' exprList? ')'
       | ID
       | '(' expr ')'
       ;
```

Поскольку `ID '(' ...` длиннее, чем просто `ID`, парсер выберет вызов, если за `ID` следует `'('`.

#### Вариант B: Объединение в одно правило

Иногда удобнее обрабатывать оба случая в одном узле дерева:

```antlr
primary: ID ('(' exprList? ')')? ;
```

Затем в `Visitor` или `Listener` проверять наличие скобок:

```csharp
if (context.exprList() != null)
else
```

Это **устраняет неоднозначность полностью**, так как нет конкурирующих альтернатив.
